# JT809 协议规范

## 1. 协议概述

### 1.1 标准信息
- **标准编号**: JT/T 809
- **标准名称**: 道路运输车辆卫星定位系统 平台间数据交换
- **当前版本**: JT/T 809-2019
- **适用范围**: 道路运输车辆卫星定位系统平台间的数据交换

### 1.2 协议特点
- 基于TCP/IP协议
- 采用全双工通信
- 支持加密传输
- 支持消息分片

## 2. 消息结构

### 2.1 消息基本格式

```
| 起始符 | 消息头 | 消息体 | CRC校验 | 结束符 |
|  1字节 | 22字节 | N字节  |  2字节  |  1字节 |
```

### 2.2 消息头结构 (22字节)

```
字节序   字段名称            类型       说明
0-3     消息长度            DWORD     整个消息长度(包括起始和结束符)
4-7     消息流水号          DWORD     按发送顺序从0开始循环累加
8-9     消息ID              WORD      标识消息类型
10-17   下级平台接入码       BCD[8]    下级平台唯一标识(行政区划码)
18-19   协议版本号标识       BYTE[2]   依次为高位字节和低位字节
20-23   上级平台接入码       DWORD     上级平台唯一标识
24-27   加密方式            BYTE      0:不加密 1:加密
28-31   加密密钥            DWORD     加密密钥(不加密时为0)
```

### 2.3 转义处理

协议中需要对特定字节进行转义，避免与标识符冲突：

**编码时转义规则**:
- `0x5b` → `0x5a 0x01`
- `0x5a` → `0x5a 0x02`
- `0x5d` → `0x5e 0x01`
- `0x5e` → `0x5e 0x02`

**解码时反转义规则**:
- `0x5a 0x01` → `0x5b`
- `0x5a 0x02` → `0x5a`
- `0x5e 0x01` → `0x5d`
- `0x5e 0x02` → `0x5e`

### 2.4 CRC校验

- **算法**: CRC-CCITT (XModem)
- **多项式**: 0x1021
- **初始值**: 0xFFFF
- **校验范围**: 从消息头到CRC校验码之前的数据

## 5. 加密机制

### 5.1 M1加密算法

JT809协议使用M1加密算法对消息体进行加密。

**加密步骤**:
1. 将加密密钥按字节拆分
2. 对消息体的每个字节进行异或运算
3. 生成加密后的数据

**示例代码**:
```java
public class M1EncryptUtil {
    /**
     * M1加密/解密
     * @param data 原始数据
     * @param key 加密密钥
     * @return 加密/解密后的数据
     */
    public static byte[] m1Encrypt(byte[] data, long key) {
        byte[] result = new byte[data.length];
        byte[] keyBytes = new byte[4];
        
        // 将密钥转换为字节数组
        keyBytes[0] = (byte) (key >> 24);
        keyBytes[1] = (byte) (key >> 16);
        keyBytes[2] = (byte) (key >> 8);
        keyBytes[3] = (byte) key;
        
        // 逐字节异或
        for (int i = 0; i < data.length; i++) {
            result[i] = (byte) (data[i] ^ keyBytes[i % 4]);
        }
        
        return result;
    }
}
```

## 7. 消息编码示例

### 7.1 上行连接请求消息示例

**原始数据**:
```
用户ID: 0x000F4240 (1000000)
密码: "12345678"
下级平台IP: "192.168.1.100"
下级平台端口: 9000 (0x00002328)
```

**编码过程**:
1. 构造消息体:
```
00 0F 42 40              // 用户ID
31 32 33 34 35 36 37 38  // 密码
31 39 32 2E 31 36 38 2E 31 2E 31 30 30 00 00 ... // IP(补零到32字节)
00 00 23 28              // 端口
```

2. 构造消息头:
```
00 00 00 36              // 消息长度(54字节)
00 00 00 01              // 流水号
10 01                    // 消息ID
01 00 00 00 00 00 00 00  // 下级平台接入码
01 00                    // 协议版本
00 00 00 01              // 上级平台接入码
00                       // 不加密
00 00 00 00              // 密钥
```

3. 计算CRC校验码

4. 添加起始符和结束符:
```
5B [消息头] [消息体] [CRC校验] 5D
```

### 7.2 转义处理示例

**原始数据**:
```
5B 10 01 5D
```

**转义后**:
```
5A 01 10 01 5E 01
```

## 8. 常见问题

### 8.1 字节序
- 所有数值类型使用大端序(Big-Endian)
- 网络传输顺序

### 8.2 BCD码
- 使用压缩BCD码
- 例如: `0x21` 表示十进制 `21`

### 8.3 字符串编码
- 默认使用GBK编码
- 固定长度字符串需要补零

### 8.4 时间格式
- BCD码格式: YYMMDDhhmmss
- 例如: `0x19 0x12 0x31 0x23 0x59 0x59` 表示 2019-12-31 23:59:59

## 9. 协议扩展

### 9.1 自定义消息
在保留消息ID范围内可以定义自定义消息类型:
- 用户自定义范围: 0x8000 ~ 0x8FFF
- 需要在上下级平台间达成一致

### 9.2 消息分片
当消息体长度超过最大传输单元时:
1. 使用消息分片机制
2. 每个分片独立CRC校验
3. 接收方重组消息

## 10. 版本差异

### 10.1 JT809-2011 vs JT809-2019

主要差异:
1. 协议版本号标识不同
2. 部分消息类型新增或调整
3. 加密算法增强
4. 支持IPv6地址

### 10.2 兼容性处理
- 通过协议版本号标识区分
- 向下兼容旧版本协议
- 根据版本号选择对应的编解码器

### 12.3 BCD编码示例

```java
/**
 * BCD编码工具类
 */
public class BCDUtil {
    /**
     * BCD编码
     */
    public static byte[] encode(String str) {
        int len = str.length();
        byte[] result = new byte[(len + 1) / 2];
        
        for (int i = 0; i < len; i++) {
            char c = str.charAt(i);
            int val = Character.digit(c, 10);
            
            if (i % 2 == 0) {
                result[i / 2] = (byte) (val << 4);
            } else {
                result[i / 2] |= val;
            }
        }
        
        return result;
    }
    
    /**
     * BCD解码
     */
    public static String decode(byte[] data) {
        StringBuilder sb = new StringBuilder();
        
        for (byte b : data) {
            sb.append(String.format("%02X", b));
        }
        
        return sb.toString();
    }
}
```
